Here is the detailed implementation plan for UniSearch backend.

The current UI should stay stable and only replace the dummy provider later. 

# UniSearch Backend Plan

## Core Principle

Do **not** build breadcrumbs first.

Build verified layers:

```text
Data acquired
↓
Data tested
↓
Data validated
↓
Next layer
↓
Final breadcrumb
```

Breadcrumbs are the final presentation, not the source data.

---

# Final Architecture

```text
unisearch/
    __init__.py

    core/
        models.py
        search_result.py
        tokenizer.py
        ranking.py

    providers/
        rna_provider.py
        operator_provider.py
        addon_preferences_provider.py
        keymap_provider.py
        ast_provider.py
        runtime_provider.py

    index/
        feature_index.py
        location_index.py
        container_index.py
        alias_index.py
        token_index.py

    resolver/
        location_resolver.py
        mode_resolver.py
        breadcrumb_builder.py

    cache/
        cache_manager.py
        fingerprint.py

    debug/
        debug_panel.py
        debug_operators.py

    ui/
        existing_panel.py
```

The current approved UniSearch panel remains the main production UI. The prototype already has a replaceable dummy provider, so the real backend can replace `get_dummy_results(query)` without redesigning the panel.

    "feature_id": "rna:View3DShading.show_cavity",

{
    "feature_id": "rna:View3DShading.show_cavity",
    "container_id": "VIEW3D_PT_shading",

# Critical Warnings

Do not start with cache.

Do not start with runtime monkey-patching.

Do not start with perfect breadcrumbs.

Do not pollute the approved UI with debug controls.

Do not promise exact mode unless the system can prove it.

The safest path is:

```text
Find feature
Validate feature
Find container
Validate container
Resolve context
Validate context
Build breadcrumb
Validate breadcrumb
Cache only after correctness
```

This gives us a manageable, testable implementation path.
